# 画面レシピ

このページでは、画面種別ごとに標準構成を固定します。個別画面の見た目を考える前に、まずどのレシピに寄せるべきかを判断してください。

## 編集画面

標準構成:

1. 戻るリンク
2. ページヒーロー
3. エラーまたは警告
4. 入力カード群
5. 送信 CTA

ルール:

- 入力カードは幅とトーンをそろえる
- 送信 CTA は右下または末尾に集約する
- フォームと無関係な補助情報は後ろへ送る
- モバイルでは主操作がファーストビューに収まる密度を優先する

## 一覧画面

標準構成:

1. ページヒーロー
2. 件数サマリー
3. フィルター
4. 一覧本体

ルール:

- フィルター UI は入力欄のトーンを統一する
- サマリーカードは補助情報として扱い、一覧より目立たせすぎない
- 行デザインはカードと競合させず、情報密度を安定させる
- モバイルで横スクロールを許容するのは、表やカルーセルのように意味がある場合だけに限定する

## 詳細画面

標準構成:

1. 戻るリンク
2. ページヒーロー
3. 主情報カード
4. コメントまたは履歴カード

ルール:

- 主役は内容確認であり、CTA ではない
- コメント、履歴、承認フローは独立カードに分ける
- 危険操作だけ色で明示し、それ以外の操作は主情報を邪魔しない位置に置く

## ダイアログ内フォーム

標準構成:

1. タイトル
2. 必要最小限の説明
3. 入力
4. 補足エラー
5. 確定 / キャンセル

ルール:

- ダイアログの中でさらに複雑なレイアウトを作らない
- 入力数が多い場合はページ遷移を検討する
- 主要 CTA とキャンセルの優先度差を明確にする

## モバイルとデスクトップの使い分け

- モバイルでは縦に素直に積み、複数の役割を 1 行に詰め込みすぎない
- デスクトップでは余白と読みやすさを増やすが、カードを巨大化しない
- 2 カラムは、主情報と補助情報の役割が明確に分かれる場合だけに使う

## レシピから外れてよいケース

次の条件を満たす場合だけ、標準レシピからの逸脱を検討します。

- 既存レシピでは業務上の優先順位を正しく表現できない
- 既存 primitive や variant 追加では解決しない
- レビュー時に逸脱理由を具体的に説明できる
