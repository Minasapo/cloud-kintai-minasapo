# 実装ルールとレビュー観点

## 実装時の判断順

新しい UI を作るときは、次の順に判断します。

1. 既存 component で表現できるか
2. 既存 component に variant を足せば済むか
3. component 専用 token を足せば済むか
4. それでも足りない場合だけ新しい primitive を検討する

## 共通ルール

- 既存 token を優先し、直接値を増やさない
- 既存 primitive を優先し、新規 UI は variant 追加で吸収する
- 見出しは `Heading` を優先し、画面ごとの手書き見出しを増やさない
- 見出しの意味階層は `level`、見た目の強さは `appearance` で調整する
- カード内や補助領域では `appearance="quiet"` を優先し、ライン装飾を持ち込まない
- 装飾ラベルは原則追加しない
- モバイルでは主操作をファーストビューに収める
- デスクトップでは最大幅を設け、広げすぎない
- 主操作、警告、補助情報の優先度差を崩さない
- 親レイアウトが子要素の横幅に引っ張られないようにする
- モバイルで 1 行に収まらない要素は、縮めるより段を分ける
- 再利用できる見た目ロジックは feature に閉じ込めず shared に戻す

## 禁止事項

- 新しい色相、影、角丸の種類をむやみに増やす
- 画面ごとに CTA の形、高さ、文字サイズを大きく変える
- 戻る導線や一覧導線を主ボタンにする
- 補助情報を主情報より目立たせる
- 1 画面内でカード種別を増やしすぎる

## レビュー時の確認項目

- 既存画面と比べてボタンのトーンがずれていないか
- 入力欄の枠線、背景色、角丸が既存とそろっているか
- ページヒーローの見出し密度が既存画面と大きく離れていないか
- 見出しの役割に応じて `Heading` の `level / appearance`、または互換ラッパーを使い分けているか
- 見出しの主役感をラインではなくサイズ、太さ、余白で出せているか
- 補助情報が主役より目立っていないか
- モバイルで左右バランスが崩れていないか
- 横スクロールや右端欠けが発生していないか

## 崩れたときの修正順

見た目がぶれた場合は、次の順で戻します。

1. レイアウト構造を既存レシピに戻す
2. ボタンの variant を標準に戻す
3. 見出しの `appearance` を `quiet` または `standard` に戻す
4. 入力欄の見た目を標準に戻す
5. カード背景、境界線、角丸を標準に戻す
6. 装飾ラベルや余計な説明を削る

## 実装前後の確認

実装前:

- `src/shared/designSystem/` に既存 token があるか確認する
- `PageSection` など shared UI で土台を流用できるか確認する

実装後:

- `/preview/design-tokens` で token の想定を確認する
- 既存画面と並べてトーンの差がないかを見る
- モバイル幅で CTA、タブ、ヘッダーが押し合っていないかを見る

## チーム運用の前提

- レビュー指摘は「雰囲気が違う」ではなく、どのルールから外れているかで行う
- 恒久ルールを更新するときは、ドキュメントサイトを先に直し、必要なら実装を追従させる
- 補助資料や検討メモを残す場合でも、最終判断はこのセクションに集約する
