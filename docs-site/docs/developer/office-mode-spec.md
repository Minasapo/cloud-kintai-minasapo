# オフィスモード / QR 打刻仕様

このページは、`/office`、`/office/qr`、`/office/qr/register` に関わる特殊運用の実装仕様を整理するための開発者向け資料です。

## 対象ルート

まずは 3 つの route がそれぞれ別の役割を持っていることを前提に読みます。`/office` は入口、`/office/qr` は表示端末、`/office/qr/register` は登録端末で使うため、画面をまたいだ変更は片側だけ直しても整合しません。

| ルート | 役割 |
| --- | --- |
| `/office` | オフィス入口。実装上は QR 表示画面へ遷移する |
| `/office/qr` | QR 表示画面 |
| `/office/qr/register` | QR 読み取り後の登録画面 |

## 設定依存

この機能は route が存在していても、AppConfig の状態によって全面的に無効化されます。単にボタンを隠す話ではなく、表示画面と登録画面の両方がガード対象になるため、設定変更時は両画面をセットで見る必要があります。

- AppConfig の `officeMode` が `true` のときだけ利用可能
- 無効時は QR 表示画面、登録画面ともに警告表示に切り替える

## QR 表示画面の仕様

表示画面側の責務は、登録に必要な最小情報を有効期限付きで配ることです。見た目の変更であっても、`mode` や token 再生成の扱いに影響すると登録画面側の前提が崩れるため、UI とロジックを分けて確認します。

- QR は `timestamp`、署名付き `token`、`mode` を含む URL を表示する
- `mode` は `clock_in` と `clock_out` をトグルで切り替える
- QR の有効期間は 30 秒
- 表示中は progress bar と残り時間を更新する
- 手動更新と URL コピーが可能
- 画面が再表示されたときも QR を再生成する

## QR 登録画面の仕様

登録画面側では、表示画面で生成した情報を信頼しすぎず、受け取った値を都度検証してから打刻処理へ進めます。ここでのバリデーションは、誤操作防止だけでなく URL 直叩きや期限切れ利用のガードも兼ねています。

- `timestamp` と `token` を検証してから登録させる
- `mode` が `clock_in` なら出勤ボタン、`clock_out` なら退勤ボタンだけを表示する
- 無効トークン、不正 mode、officeMode 無効時はエラー表示にする
- 登録処理は既存の勤怠 upsert / update 系処理を利用する

## 打刻登録の流れ

実際の登録処理は専用 API を新設するのではなく、既存の勤怠更新フローへ乗せています。そのため、ここを変えると通常打刻と同じ更新系ロジックやキャッシュ反映の挙動にも影響しうる前提で見てください。

1. 現在時刻から業務日を解決する
2. `clockInAction` または `clockOutAction` を呼ぶ
3. idempotency key を生成して mutation を実行する
4. 成功時は通知を出し、取得済み attendance を更新する

## 現在のドキュメント上の扱い

- 利用者向け正本: [オフィスモードを運用する](../admin/office-mode-operations)
- 画面索引上は `特殊運用` セクションに分類する
- 独立ロールとして docs を増やさない

## 変更時の確認観点

この機能は route、AppConfig、トークン生成、利用者向けガイドが密結合しています。変更点が 1 か所に見えても、実際には画面表示、登録条件、ドキュメント運用をまとめて更新する必要があります。

- QR 有効期限を変える場合
  - 表示 UI、登録画面のトークン検証、運用ガイドを同時に更新する
- mode を増やす場合
  - `screen-list`、利用者向けガイド、登録 UI、token 付き URL 生成を更新する
- officeMode のガード条件を変える場合
  - AppConfig 取得箇所と disabled 表示の整合を確認する
