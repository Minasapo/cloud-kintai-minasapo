# 打刻補助設定と表示依存

このページは、打刻画面や勤怠修正入力を補助する AppConfig 依存設定の実装仕様を整理するための開発者向け資料です。

## 対象設定

対象設定はいずれも AppConfig で切り替わりますが、実際の参照箇所は 1 画面に閉じません。設定名だけを追うのではなく、「どの画面で」「どの条件で」読まれるかを意識して確認すると影響範囲をつかみやすくなります。

- 打刻画面アナウンス
- 外部リンク
- 打刻理由
- クイック入力
- ワークフロー通知機能

## 打刻画面アナウンス

アナウンスは単純な表示フラグではなく、既読制御まで含めた UI 挙動を持っています。文面変更で再表示される設計のため、設定値の更新は見た目以上に利用者体験へ影響します。

- `timeRecorderAnnouncementEnabled` と `timeRecorderAnnouncementMessage` に依存する
- 表示条件は `enabled` かつ `message.trim().length > 0`
- 閉じた状態は `configId` と本文をキーに localStorage へ保存する
- 本文が変わると別キー扱いになり、再度表示される

## 外部リンク

外部リンクは設定値としては配列ですが、実際には URL 展開や表示アイコンなど複数の表現層にまたがります。単に項目を追加できるかだけでなく、利用側の整形や表示崩れも合わせて確認する前提です。

- AppConfig の `links` 配列を参照する
- 各リンクは `label`、`url`、`enabled`、`icon` を持つ
- URL 内の `{staffName}` は利用側で展開される前提

## 打刻理由

打刻理由は、入力補助の見え方と申請内容の読みやすさの両方に効きます。理由候補の追加や無効化は API 形状よりも UI の使い勝手に影響しやすいため、利用画面をまたいで確認します。

- AppConfig の `reasons` 配列を参照する
- enabled な理由だけが勤怠修正画面の入力補助に出る
- スタッフ向けと管理者向けの修正 UI で利用箇所を確認する

## クイック入力

クイック入力は参照箇所が複数あるため、1 か所で正しく見えても全体が安全とは限りません。特にスタッフ向けと管理者向けで UI 密度が違うので、候補数や表示順の変更は複数画面で見比べる必要があります。

- `quickInputStartTimes` と `quickInputEndTimes` を参照する
- enabled な候補だけがボタンとして表示される
- スタッフ向け、管理者向け、モバイル向けで参照箇所が複数ある

## ワークフロー通知機能

通知機能は単独の表示設定ではなく、通知一覧画面、ヘッダー導線、URL 生成までを含む機能単位で存在します。現在は管理者向け URL 前提に古い導線が残っているため、設定変更時ほど route との整合確認が重要です。

- `workflowNotificationEnabled` が `true` のときだけ通知アイコンと通知一覧導線を表示する
- `WorkflowNotificationsPage` 自体も無効時は別画面へリダイレクトする
- 現在のコードでは管理者通知が `/admin/workflow` を前提にした URL を組み立てるため、router との整合に注意が必要

## 変更時の確認観点

この節は、設定値そのものより「更新をどこまで波及させるか」を確認するためのチェックリストです。AppConfig を触る変更は、設定 UI、表示側、docs の三者がそろって初めて完了と考えます。

- AppConfig 項目を追加、改名、削除する場合
  - 設定 UI、利用画面、利用者向け正本を同時に更新する
- 通知導線を変更する場合
  - `WorkflowNotificationsPage` の遷移先と router 定義を合わせる
- 打刻補助設定を増やす場合
  - `settings-item-list` と個別運用ガイドの両方を更新する
