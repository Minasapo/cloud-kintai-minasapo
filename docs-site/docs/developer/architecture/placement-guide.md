# 配置判断ガイド

このページは、実装時に「どこへ置くべきか」を素早く判断するための実務ガイドです。

## まず決めること

次の順で判断すると、配置の迷いを減らせます。

1. そのコードは特定ページ専用か。
2. そのコードは機能単位で再利用するか。
3. そのコードはドメイン知識を含むか。
4. そのコードはアプリ全体で汎用的に使うか。

## 配置の目安

| 迷った対象                             | 推奨配置          | 理由                               |
| -------------------------------------- | ----------------- | ---------------------------------- |
| 画面コンポーネント                     | `src/pages/*`     | ルーティング単位で管理するため     |
| 複数ページをまたぐ導線                 | `src/processes/*` | 業務フローを一箇所に集約できるため |
| 画面内機能（フォーム、一覧、フィルタ） | `src/features/*`  | 機能ごとに独立して保守しやすいため |
| 型、ドメイン判定、API クライアント     | `src/entities/*`  | ドメイン責務を明確に分離できるため |
| 汎用 UI、共通 hook、ユーティリティ     | `src/shared/*`    | 横断再利用と一貫性を保てるため     |
| ページ大型ブロック（ヘッダー等）       | `src/widgets/*`   | レイアウト単位で再利用できるため   |

## よくある配置ミス

- 画面専用コンポーネントを `shared/ui` に置いてしまう
- 特定ドメイン向け hook を `shared/hooks` に置いてしまう
- API 呼び出しと画面表示ロジックを同じ層に混在させる
- 一時実装を `pages` に残したまま機能分離しない

## 迷ったときの優先順位

1. 依存方向を壊さない
2. 将来の再利用範囲を見積もる
3. ドメイン知識を `entities` に寄せる
4. UI とロジックの責務を分離する

## 最小チェックリスト

- 依存は `pages → processes → features → entities → shared` を守っている
- `shared` に画面固有の責務が混ざっていない
- 自動生成領域を手編集していない
- 新規追加した場所を README / docs から辿れる
